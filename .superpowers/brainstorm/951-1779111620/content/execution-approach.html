How should each phase be executed internally?

The 6-phase structure is locked. Now we decide how Claude Code builds within each phase — this determines speed, safety, and coordination overhead.

A

Sequential Safest

Every milestone built one at a time in the main Claude Code session. Simple, no coordination overhead, full context carried throughout.

Plugins used inline: context7 for library docs as needed, sequential-thinking for complex logic like the milestone template engine, frontend-design before each UI milestone.

Pros

  • No coordination complexity
  • Full context always available
  • Easiest to review and redirect

Cons

  • Slowest — no parallel work
  • Context window grows large by Phase 4+
B

Parallel Subagents Within Phases Recommended

Foundation milestones (M0–M2) built sequentially — everything depends on them. Once the schema and shell exist, independent milestones within a phase run as parallel subagents, then merge before the review gate.

Plugin strategy: Main session uses sequential-thinking for architecture decisions + context7 for library lookups. Subagents use frontend-design for UI milestones. The dispatching-parallel-agents superpowers skill orchestrates this.

Pros

  • Significantly faster on middle phases
  • Each subagent gets a clean context window
  • Matches how the platform modules relate to each other

Cons

  • Requires careful task handoff briefs to subagents
  • Convention drift risk — mitigated by shared CLAUDE.md
C

Worktree Isolation Per Milestone Cleanest Git History

Each milestone gets its own git branch + worktree. Work is isolated, reviewed as a PR-style diff at each gate, then merged. Matches the plan's "one milestone = one PR" principle exactly.

main branch
→ worktree →
feat/m0-foundation
→ merge →
main
→ worktree →
feat/m1-data
→ ...

Review gates become literal code reviews — you can see exactly what changed in each milestone. Plugins work the same as Option A.

Pros

  • Clean, auditable git history
  • Review gates have a clear "diff" to inspect
  • Easiest to revert a single milestone

Cons

  • Git overhead — no repo yet, must init first
  • Slower than parallel subagents
  • Merge conflicts at phase boundaries
